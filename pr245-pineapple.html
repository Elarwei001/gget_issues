4数据托管在哪 / 为何硬编码Where the data lives / why hard-coded

所有资源都是 Google Drive 上的公开文件，每个对应一个文件 ID。下载 URL 形如：All resources are public files on Google Drive, each mapped to a file ID. The download URL looks like:

https://drive.google.com/uc?export=download&id=<FILE_ID>

无鉴权、无 OAuth。但有一个关键行为——病毒扫描确认页：当文件大到 Google 无法扫描（约 >100MB）时，它不直接返回文件，而是先返回一个 HTML 页面（"Google Drive can't scan this file for viruses"），带一个需要提交的表单（含 confirm token）。客户端必须解析并再次提交，才能拿到真正字节。gget 必须处理这一步，否则大文件会把 HTML 警告页存成损坏的 .tar.gz。No authentication, no OAuth. But there is one key behavior — the virus-scan confirmation page: when a file is too large for Google to scan (roughly >100MB), it does not return the file directly, but first returns an HTML page ("Google Drive can't scan this file for viruses") with a form that must be submitted (containing a confirm token). The client must parse and re-submit it before getting the real bytes. gget must handle this step, otherwise large files will save the HTML warning page as a corrupt .tar.gz.

为何硬编码这 30 个 ID？ 上游自己就是在 Rust 源码里逐个硬编码的，作者在注释里也承认"不是好做法、以后可能挪出去"。因为没有可编程 API、也没有目录式 manifest，gget 只能镜像成一份时间点快照。决策：v1 硬编码 + live 测试兜底 + 维护者 note 说明。（session 中还写过一个"运行时解析上游 .rs 拿 ID 表"的原型，作为将来防腐烂的可选路径，见 §9/§11。）Why hard-code these 30 IDs? Upstream itself hard-codes them one by one in the Rust source, and the author even admits in a comment that it's "not a good practice, may be moved out later." Because there is no programmatic API and no manifest-style directory, gget can only mirror a point-in-time snapshot. Decision: v1 hard-code + live test as a safety net + a maintainer note explaining it. (During the session a prototype that "parses the upstream .rs at runtime to obtain the ID table" was also written, as an optional future anti-rot path — see §9/§11.)

7Live data 测试如何设计How the live-data test is designed

难点：archs4/opentargets 的 live 测试是"真调用→比对返回值"。但 pineapple 的返回值只是本地目录表，真正的"数据"是 GB 级 Google Drive 大文件——CI 不能真下载。所以"live"体现在：对已知资源真实请求 Google Drive，验证它仍解析得到可下载文件，但只读 header、不取 body。The challenge: the archs4/opentargets live tests are "make a real call → compare the return value." But pineapple's return value is just a local catalog table, and the real "data" is GB-scale Google Drive files — CI cannot actually download them. So "live" here means: make a real request to Google Drive for a known resource and verify it still resolves to a downloadable file, but read only the header, not the body.

9关键工程决策与权衡Key engineering decisions and trade-offs

① 硬编码镜像 vs 运行时解析① Hard-coded mirror vs runtime parsing

v1 选硬编码（上游无 API/manifest，且上游自己也硬编码）。防腐烂手段：live 测试兜底 + 维护者 note。原型 parse_pineapple_registry.py 验证过"运行时解析上游 .rs 拿 ID 表"可行（在线跑通、30 项数量与 live 校验全对），作为将来可选升级——好处是上游一改自动跟上、不再腐烂，且不引入 Rust 二进制。v1 chose hard-coding (upstream has no API/manifest, and upstream itself hard-codes). Anti-rot measures: live test as a safety net + a maintainer note. The prototype parse_pineapple_registry.py verified that "parsing the upstream .rs at runtime to obtain the ID table" is feasible (ran online, the 30-item count and live checks all matched), as an optional future upgrade — the benefit being that it automatically follows upstream changes, no longer rots, and introduces no Rust binary.

② category 用 Literal，不用 enum② category uses Literal, not enum

gget 全库公共 API 无 enum、且是双接口（CLI 只认字符串）。最终给 category 加 Literal["segmentation","benchmark","weights"]（gget 本身惯用法）→ 拿到编辑器/类型检查的静态容错，又不破坏一致性、不影响 CLI。name（30 个、会变的镜像）保持带友好报错的自由字符串。Across gget's public API there is no enum, and it is a dual interface (the CLI only accepts strings). In the end category was given Literal["segmentation","benchmark","weights"] (gget's own idiom) → gaining editor/type-checker static safety without breaking consistency or affecting the CLI. name (30 items, a changing mirror) stays a free string with friendly error messages.

③ 为什么不"一键装 Rust CLI"③ Why not "one-click install the Rust CLI"

会破坏 gget 纯 Python/pip 的架构、抬高门槛（cargo 需编译）、跨平台二进制负担、且自相矛盾（PR 卖点就是"免装 Rust"）。数据本就在公开 Google Drive，gget 直接当 HTTP 客户端取即可。It would break gget's pure-Python/pip architecture, raise the barrier to entry (cargo requires compilation), impose cross-platform binary burdens, and be self-contradictory (the PR's selling point is "no Rust install needed"). The data is already on public Google Drive, so gget can just act as an HTTP client and fetch it.

④ 可复现性 —— agent 时代库的价值④ Reproducibility — the value of libraries in the agent era

讨论过"有 agent 后 gget 还有什么用"。结论：薄封装（如 pineapple）的边际价值确实被侵蚀；但可复现性（固定版本、确定输出、可引用/可写进论文方法）、规模化/流水线、沉淀+测试过的领域知识是 agent 短期替代不了的。最佳形态是 agent 调用可信的库——库不消失，重新定位为 agent 依赖的确定性原语。We discussed "what use is gget once agents exist." Conclusion: the marginal value of thin wrappers (like pineapple) is indeed eroded; but reproducibility (pinned versions, deterministic output, citable / writable into a paper's methods), scaling/pipelines, and codified + tested domain knowledge are things agents cannot replace in the short term. The best shape is an agent calling a trusted library — the library does not disappear but is repositioned as a deterministic primitive the agent relies on.
