TL;DR：Pineapple 是个 Rust 命令行图像分析工具，同时策展了一批标准化的生物影像数据集 + 预训练权重，托管在公开 Google Drive 上，但没有 REST/JSON API。这个 PR 给 gget 加了一个 pineapple 模块，把这批数据的目录列举和按名下载封装进来——用户无需安装 Rust 就能用；只做数据下载，不碰 Pineapple 的图像处理功能。本次还补了一套 live data 测试（抽样打真实 Google Drive、只读 header）保证数据长期可访问，以及 en/es 双语文档。

2这个 PR 解决什么问题

需求来自 issue #161，标题就是 "Pineapple module to download bio-imaging data"，由 gget 维护者 Laura Luebbert 提出，并注明 "suggested by Prof. Anne Carpenter"（Carpenter 是 image-based profiling / CellProfiler 领域的奠基人物）。所以需求从第一天起就钉死在"下载生物影像数据"，不是图像处理。

核心洞察（也写进了 PR 描述）：

Pineapple 是个 Rust CLI，不暴露 REST/JSON API。它通过 pineapple download 提供的，是一份策展、标准化的数据集 + 预训练权重目录，托管在 Google Drive——每个资源在 pineapple-data crate 里对应一个硬编码的 Google Drive 文件 ID。gget 这个模块镜像了这份目录，直接从同一个 Google Drive 源下载，让用户不装 Rust 二进制就能拿到数据。

3上游 Pineapple 是什么

Pineapple（作者 Tom Ouellette，BSD-3 许可）是用 Rust 写的基于图像的细胞分析（image-based cell profiling）命令行工具，能做：分割对象抽取、计算形态学描述子、生成细胞/细胞核的自监督 embedding 等。它是个大得多的工具，子命令包括：

子命令作用gget 是否封装
process从图像+mask/polygon/box 抽取对象级数据否
profile计算对象级形态学描述子否
neural自监督 embedding（神经网络）否
measure / utils度量 / 工具否
download列目录 + 下载策展数据集/权重✅ 就封装这一个

注：仓库里没有论文/preprint（无 DOI），引用即 GitHub repo 本身。gget 的定位是数据访问而非图像分析，所以只镜像 download 是正确的 scoping。

4数据托管在哪 / 为何硬编码

所有资源都是 Google Drive 上的公开文件，每个对应一个文件 ID。下载 URL 形如：

https://drive.google.com/uc?export=download&id=<FILE_ID>

无鉴权、无 OAuth。但有一个关键行为——病毒扫描确认页：当文件大到 Google 无法扫描（约 >100MB）时，它不直接返回文件，而是先返回一个 HTML 页面（"Google Drive can't scan this file for viruses"），带一个需要提交的表单（含 confirm token）。客户端必须解析并再次提交，才能拿到真正字节。gget 必须处理这一步，否则大文件会把 HTML 警告页存成损坏的 .tar.gz。

为何硬编码这 30 个 ID？ 上游自己就是在 Rust 源码里逐个硬编码的，作者在注释里也承认"不是好做法、以后可能挪出去"。因为没有可编程 API、也没有目录式 manifest，gget 只能镜像成一份时间点快照。决策：v1 硬编码 + live 测试兜底 + 维护者 note 说明。（session 中还写过一个"运行时解析上游 .rs 拿 ID 表"的原型，作为将来防腐烂的可选路径，见 §9/§11。）

5接入 gget 后如何使用

关键认知：pineapple() 的返回值永远是"元数据表"（DataFrame / json）；压缩包只是 download=True 时写到磁盘的副作用，不通过返回值传。用户场景是两段式的：先看菜单 → 再下单个。

三种输入 → 输出

输入返回值副作用
gget.pineapple(category="segmentation")
不给 name		整张目录表（该类全部资源的元数据）无
gget.pineapple("vicar_2021")
给 name、不下载		单行元数据表无
gget.pineapple("vicar_2021", download=True, out_dir="./d")仍是那行元数据表把 vicar-2021.tar.gz 存到 ./d/

命令行 & Python 对照

# CLI —— 列目录
gget pineapple --category segmentation
# CLI —— 按名下载
gget pineapple vicar_2021 --download --out_dir ./pineapple_data
# CLI —— 列预训练权重
gget pineapple --category weights
import gget
gget.pineapple(category="segmentation")                 # 列目录 -> DataFrame
gget.pineapple("vicar_2021", json=True)                 # 单个元数据 -> list[dict]
gget.pineapple("vicar_2021", download=True, out_dir="./pineapple_data")  # 下载 + 返回元数据

目录（catalog）字段

列含义备注

文件名两套派生规则：数据集 = name.replace("_","-") + ".tar.gz"；权重 = 独立的显式 .safetensors 文件名。

6Scope：只做下载

模块刻意只封装 Pineapple 的数据分发层（pineapple download）：列目录 + 按名从 Google Drive 下载。不复制 process/profile/neural/measure 这些图像处理能力——那是 Pineapple 工具本体的核心，超出 gget（数据访问而非图像分析）的范围。想处理图像的用户应直接用 Pineapple。这与 issue #161 的原始诉求（"download bio-imaging data"）一致，并已写进代码/文档/PR 描述。

7Live data 测试如何设计

难点：archs4/opentargets 的 live 测试是"真调用→比对返回值"。但 pineapple 的返回值只是本地目录表，真正的"数据"是 GB 级 Google Drive 大文件——CI 不能真下载。所以"live"体现在：对已知资源真实请求 Google Drive，验证它仍解析得到可下载文件，但只读 header、不取 body。

关键实测发现（先探针、再写断言）

对真实 ID 探 header 发现两个断言级信息，以及一个救命的反例：

  • Content-Disposition 带真实文件名 → Google Drive 亲口告诉你这个 ID 现在服务的是哪个文件，是最精确的身份锚。
  • Content-Length = 精确字节数；Content-Type: application/octet-stream 证明是二进制、非报错 HTML。
  • 反例：livecell 目录标 3.26GB，实际 Content-Length 只有 1.81GB → 说明 size_gb 不可靠，size 断言必须放宽，不能紧绑目录值。
  • license / author 不在任何 header 里 → 无法对真值校验，只能信任上游镜像。

最终断言（每个抽样资源）

断言守护的回归
HTTP 200ID 还在、文件没被删
Content-Type 含 octet-stream（非 html）没退化成配额/报错页
Content-Disposition 文件名 == 目录 filename（精确）ID 没被改指到别的文件
Content-Length > 1MB（宽松，不绑 size_gb）ID 没被换成占位小文件

抽样：代表性 4 个（非全部 30）

资源覆盖点
arvidsson_2022 (seg, ~28MB).tar.gz + 直连下载路径
kromp_2023 (benchmark, ~25MB)benchmark 类
dino_vit_small (weights, ~88MB)显式 .safetensors 文件名规则
hpa_2022 (seg, ~1.6GB)触发病毒扫描确认页分支（header-only，不下 body）

抗 flaky：限流 → skip

Google Drive 短时间被请求太多会返回"配额超限"的 HTML 页（临时限流）。测试区分两种"拿不到"：数据真没了（404/410 或文件名对不上）→ 硬失败（CI 红）；Google 临时限流（返回 html）→ skip（CI 绿）。即"if Google Drive temporarily rate-limits the request, that resource is skipped rather than failing the build"。

复用生产代码路径 + 负向验证

  • 把下载里"GET→判 html→解析病毒扫描 form→确认"的解析流程抽成 _resolve_gdrive_response()，下载与测试共用 → live 测试走的是和真实下载完全相同的生产路径。
  • 做过负向验证：故意喂错误的预期文件名 → 断言确实 fail，证明测试能真正抓到"ID 被改指/改名"，不是空测。
  • 耗时：抽样 4 个约 17s；曾试过全量 30 个约 96s，为 CI 速度与稳定性最终选 4 个（全量可作 env-gate，见 §11）。

9关键工程决策与权衡

① 硬编码镜像 vs 运行时解析

v1 选硬编码（上游无 API/manifest，且上游自己也硬编码）。防腐烂手段：live 测试兜底 + 维护者 note。原型 parse_pineapple_registry.py 验证过"运行时解析上游 .rs 拿 ID 表"可行（在线跑通、30 项数量与 live 校验全对），作为将来可选升级——好处是上游一改自动跟上、不再腐烂，且不引入 Rust 二进制。

② category 用 Literal，不用 enum

gget 全库公共 API 无 enum、且是双接口（CLI 只认字符串）。最终给 category 加 Literal["segmentation","benchmark","weights"]（gget 本身惯用法）→ 拿到编辑器/类型检查的静态容错，又不破坏一致性、不影响 CLI。name（30 个、会变的镜像）保持带友好报错的自由字符串。

③ 为什么不"一键装 Rust CLI"

会破坏 gget 纯 Python/pip 的架构、抬高门槛（cargo 需编译）、跨平台二进制负担、且自相矛盾（PR 卖点就是"免装 Rust"）。数据本就在公开 Google Drive，gget 直接当 HTTP 客户端取即可。

④ 可复现性 —— agent 时代库的价值

讨论过"有 agent 后 gget 还有什么用"。结论：薄封装（如 pineapple）的边际价值确实被侵蚀；但可复现性（固定版本、确定输出、可引用/可写进论文方法）、规模化/流水线、沉淀+测试过的领域知识是 agent 短期替代不了的。最佳形态是 agent 调用可信的库——库不消失，重新定位为 agent 依赖的确定性原语。
