TL;DR：PR #242 新增 gget mitocarta 模块，把 Broad Institute 的 MitoCarta3.0（哺乳动物线粒体蛋白与通路的人工策展清单）从上游托管的两个 .xls Excel 数据库（人/鼠；这只是官网整批发布文件中的一个子集，另有 .gmx/.bed/.fasta/图像等）下载下来，解析成 pandas DataFrame / JSON，支持 human/mouse 与每个 .xls 内的三张表（mitocarta 清单 / all_genes 全基因 Maestro 评分 / pathways 通路层级）。这是纯数据访问（取一张已发布的表并 gget 化），没有越界重算线粒体定位——平台契合度良好，落在 gget 的「取 blob」薄封装一端（类 ref/pineapple）。主要风险是可维护性：URL 硬编码到 personal.broadinstitute.org/scalvo/ 一个个人目录（无 API/无 DOI，易腐烂），且现有测试直接打真实 URL 却硬断言精确行数 1136、无限流/网络异常 → skip，一旦上游抖动就会误红。结论：方案科学、契合、可复现（版本固定），但需要一个按 live-data 规范重写的兜底测试 + 把 xlrd 依赖考虑成可选 extra。

3上游 / 数据源（MitoCarta3.0）

MitoCarta3.0（Rath et al., NAR 2021）是 Broad Institute Mootha 实验室维护的人工策展线粒体蛋白质组清单。关键：它没有可编程 API，而是以一整批不同格式的静态文件发布在一个个人 web 目录下（见下），不是单个文件。

官网发布的文件清单（Broad 下载页）： Human.MitoCarta3.0.xls、Mouse.MitoCarta3.0.xls（两个 Excel 数据库，含 1136/1140 线粒体基因 + 亚定位 + 通路 + ~20,000 基因的整合评分）、MitoPathways3.0.gmx（149 通路，供 GSEA）、Human/Mouse .bed（UCSC 轨道）、Human/Mouse .fasta（RefSeq 蛋白序列）、GFP 定位图像、Mouse.MSMS.peptides.xlsx 等。
本 PR 只 wrap 其中两个 .xls（人/鼠 Excel 数据库），不含 .gmx / .bed / .fasta / 图像等。这个子集选择是合理的：.xls 是**规范清单本体**，而序列（gget 已有 seq）、.bed/.gmx 属特定工具窄用途。（可作为一条"未来可选扩展"记入 §未决。）

5如何使用

输入：species（human/mouse）+ which（选哪张表）。返回：CLI 默认 JSON（-csv 转 CSV）；Python 默认 DataFrame（json=True 转 list[dict]）。副作用：save/--out 落盘。

# CLI：抓取人类线粒体基因清单（CSV）
gget mitocarta --species human --csv

# CLI：抓取 MitoPathways 通路层级
gget mitocarta --which pathways --csv
# Python
import gget
df = gget.mitocarta(species="human", which="mitocarta")   # ~1136 行 DataFrame
paths = gget.mitocarta(which="pathways")                    # 通路层级表
recs = gget.mitocarta(which="all_genes", json=True)         # list[dict]

save / --outFalse / 路径落盘 CSV 或 JSON

7设计决策与权衡

① 按 sheet 名首字母（A/B/C）解析，而非硬编码全名

选了用 name.startswith("A "/"B "/"C ") 定位 sheet，理由是人/鼠工作簿里物种词不同（A Human MitoCarta3.0 vs A Mouse MitoCarta3.0）；放弃了硬编码整串 sheet 名。优点：一套代码兼容两物种。代价：若上游改动首字母命名约定即失效——但有清晰报错（列出可用 sheet），可接受。

② 内存内下载 + io.BytesIO 解析，不落临时文件

≈9.7 MiB 全量拉进内存再交给 pd.ExcelFile(..., engine="xlrd")。文件小，干净；代价：每次调用都重新下载（无缓存），重复调用略浪费。

③ 引入 xlrd>=2 运行时依赖

上游是旧式 .xls，只有 xlrd>=2（纯 Python、无二进制/编译）能读。符合 gget「纯 Python、pip 可装」的架构底线（不像 Rust/编译工具那样违背架构）。权衡：为单个模块给 core 加了一个运行时依赖；PR 作者已在描述里主动提议「可改成 optional extra」——建议采纳，与 gget[cellxgene] 的处理一致。

④ URL 硬编码到个人目录

直连 personal.broadinstitute.org/scalvo/MitoCarta3.0/*.xls。优点：版本固定 → 输出确定、可复现、可引用。最大风险：个人 web 目录无 SLA/无 DOI/无 API，一旦作者迁移或清理即 404 → 这是时间点快照式镜像，必须靠 live test 兜底发现腐烂。

8平台契合度评估（四 lens）

Lens评分依据
① Soundness
正确 / 健壮		 良好 输入校验（species/which 均 ValueError）、状态码非 200 → RuntimeError、缺 xlrd → 友好报错、sheet 找不到时列出可选项。json 参数用 json as json_package 别名避免遮蔽 stdlib。小瑕疵：无缓存、无重试/超时以外的网络弹性。
② 平台契合度
访问 vs 分析		 通过（核心结论） 纯数据访问：取一张已发布的表 → 结构化透出，无越界重算。返回值已 gget 化（DataFrame/JSON）、双接口齐备、无违背架构的二进制依赖。落在「取 blob」薄封装一端，类 ref/pineapple，合理。
③ 可复现性 良好 URL 指向固定版本（3.0，Last-Modified 2023-03）→ 确定输出，可写进论文方法、可被审稿人重跑、可引用（cite.md 已补 MitoCarta 引文）。这是「库」相对「临时 agent 调用」不可替代之处。
④ 维护成本 需关注 会腐烂：硬编码个人目录 + 依赖 sheet 命名约定 + 硬断言精确行数。腐烂如何被发现？——目前不足：现有测试直接打真实 URL 但无「限流/网络错 → skip」，上游一抖动就误红；也无 nightly/env-gate 分层。→ 见第 8 节。

9测试：mock vs live

PR 现有 tests/test_mitocarta.py + fixture tests/fixtures/test_mitocarta.json，共 4 个用例。关键观察：其中两个「属性测试」实际会真实下载上游 .xls，因此它们本质上是 live 测试，但没有按 live-data 规范设计。

层测什么现状
network-free 单元测试 本地代码正确性（参数校验、报错） 有：bad_species/bad_which 在下载前就 ValueError，离线可跑、确定性。
live data 测试
（远端可访问性）		 MitoCarta 源还在、还对 名为单元测试、实为 live：test_mitocarta_human 真下载并断言 len==1136（精确）+ 列存在 + 含 MT-CO1；pathways 断言 MitoPathway 列 + >=100 行。
按 skill 的 live-data 规范，现有测试需要三处加固：
  1. 限流/网络错 → skip 而非红：现在上游 404/超时/返回 HTML 会让 CI 硬红（误报）。应捕获 requests.RequestException 与「返回非 Excel」时 self.skipTest(...)。
  2. 精确行数 1136 太脆：应放宽为下限（如 >= 1000），把「身份锚」留给能对真值校验的断言——MT-CO1 ∈ Symbol、期望列存在、Content-Type 是 Excel 而非 HTML。这些锚已部分具备，方向对。
  3. 轻量化 + 分层：源支持 Accept-Ranges: bytes，可先用 HEAD 探 Content-Type/Content-Length 做廉价探活；全量解析放 env-gate/nightly，并明确 log「只测了 human 一物种」。
发布前应做一次负向验证（临时喂错文件名/错行数确认断言真的 fail），证明测试不是空测。

注：≈9.7 MiB 的全量下载在 CI 里尚可接受（非 GB 级），所以「只读 header」在这里是优化项而非硬性要求；但「限流→skip + 松绑精确值」是必须的，否则维护成本 lens 不达标。
