什么是「通路」？A/B 与 C 又是什么关系What is a "pathway"? And how do A/B and C relate?

📖 通路（pathway）= 细胞里的一条"流水线"A pathway = an "assembly line" inside the cell：一组蛋白/基因分工协作完成某个任务。例如 OXPHOS（氧化磷酸化）是线粒体"发电厂"的发电机组，由 Complex I~V 几台大机器组成，每台机器又由许多"零件（subunit 亚基）"拼成——每个零件就是一个基因。: a group of proteins/genes that divide labor and cooperate to accomplish a task. For example, OXPHOS (oxidative phosphorylation) is the generator set of the mitochondrial "power plant," made of the several large machines Complex I~V, and each machine is in turn assembled from many "parts (subunits)" — where each part is a gene.

A/B 里的通路记法怎么读（MitoCarta3.0_MitoPathways 列），以 NDUFS1 为例：How to read the pathway notation in A/B (the MitoCarta3.0_MitoPathways column), using NDUFS1 as an example:

NDUFS1 → OXPHOS > Complex I > CI subunits | Metabolism > Metals and cofactors > Fe-S-containing proteins | OXPHOS > OXPHOS subunits
  • > = "隶属"，像文件夹路径从大类到具体：OXPHOS > Complex I > CI subunits = 发电机 OXPHOS ▸ Complex I 这台机器 ▸ 它的一个零件。> = "belongs to," like a folder path going from the broad category to the specific: OXPHOS > Complex I > CI subunits = the OXPHOS generator ▸ the Complex I machine ▸ one of its parts.
  • | = "还兼任"，分隔一个基因的多个角色。上面 NDUFS1 身兼三职（Complex I 零件 / 含铁硫簇的蛋白 / OXPHOS 亚基）。CYC1 则是 Complex III 的零件、兼管 Heme（血红素）代谢。| = "also serves as," separating a gene's multiple roles. Above, NDUFS1 holds three roles (a Complex I part / an Fe-S-cluster-containing protein / an OXPHOS subunit). CYC1, meanwhile, is a part of Complex III and also handles Heme metabolism.

对接口设计的启示：① A/B 共 48 列且列头一致 → 可共用一套列定义；② 暴露 mitocarta / all_genes / pathways 三表、跳过 Description 合理；③ 48 列里有大量 MS/MS 与逐组织表达细列，若嫌表太宽，可考虑默认返回核心子集、加参数开全集；④ pathways 与 mitocarta/all_genes 覆盖"通路→基因"与"基因→通路"两个查询方向，都值得留；C 的 Genes 是逗号分隔长串，接口可 split 成列表更好用。Implications for interface design: (1) A/B share 48 columns with identical headers → one column definition can be shared; (2) exposing the three tables mitocarta / all_genes / pathways and skipping Description is reasonable; (3) among the 48 columns there are many detailed MS/MS and per-tissue expression columns — if the table feels too wide, consider returning a core subset by default with a parameter to open the full set; (4) pathways and mitocarta/all_genes cover both query directions, "pathway → genes" and "gene → pathways," and both are worth keeping; C's Genes is a long comma-separated string, and the interface would be more usable if it split it into a list.
✅ 接口设计定调：只做 L2「干净 tidy 表」，不做 L3 矩阵。 具体到 mitocarta，L2 = 每个 (species, which) 返回一张规范 DataFrame：✅ Interface design decision: do only L2 "clean tidy tables," not L3 matrices. Concretely for mitocarta, L2 = each (species, which) returns one normalized DataFrame:
  • 读对 sheet、修好 C 表的错位表头（别让首列变成 "2"）；read the right sheet and fix C's shifted header (don't let the first column become "2");
  • 类型规范：分数 → float、坐标 → int、列表字段 → list；normalize types: scores → float, coordinates → int, list fields → list;
  • 把分隔串拆成列表：MitoCarta3.0_MitoPathways（| 分隔）、pathways 表的 Genes（逗号分隔），必要时 Synonyms/Tissues；split delimited strings into lists: MitoCarta3.0_MitoPathways (|-separated), the Genes column of the pathways table (comma-separated), and where needed Synonyms/Tissues;
  • 基因名以字符串读入（避免 Excel 式日期损坏在下游重现）；跳过 Description sheet。read gene names as strings (to avoid Excel-style date corruption reappearing downstream); skip the Description sheet.
L3（基因×组织 / 基因×通路 矩阵）不在模块里做——用户一行 pivot/get_dummies 即可，文档给个示例、等有 request 再加。理由：接口薄而稳、测试面小，符合 gget 调性。L3 (gene × tissue / gene × pathway matrices) is not done inside the module — a user can do it in one line of pivot/get_dummies; give a docs example and add it later if there is a request. Rationale: a thin, stable interface with a small test surface, consistent with gget's style.

7设计决策与权衡Design decisions & trade-offs

① 按 sheet 名首字母（A/B/C）解析，而非硬编码全名① Parse by the sheet name's leading letter (A/B/C) rather than hard-coding the full name

选了用 name.startswith("A "/"B "/"C ") 定位 sheet，理由是人/鼠工作簿里物种词不同（A Human MitoCarta3.0 vs A Mouse MitoCarta3.0）；放弃了硬编码整串 sheet 名。优点：一套代码兼容两物种。代价：若上游改动首字母命名约定即失效——但有清晰报错（列出可用 sheet），可接受。The choice was to locate sheets via name.startswith("A "/"B "/"C "), because the species word differs between the human and mouse workbooks (A Human MitoCarta3.0 vs A Mouse MitoCarta3.0); hard-coding the full sheet-name string was rejected. Pro: one codebase works for both species. Cost: it breaks if upstream changes the leading-letter naming convention — but there is a clear error message (listing the available sheets), which is acceptable.

② 内存内下载 + io.BytesIO 解析，不落临时文件② In-memory download + io.BytesIO parsing, no temp file on disk

≈9.7 MiB 全量拉进内存再交给 pd.ExcelFile(..., engine="xlrd")。文件小，干净；代价：每次调用都重新下载（无缓存），重复调用略浪费。The full ≈9.7 MiB is pulled into memory and handed to pd.ExcelFile(..., engine="xlrd"). The file is small, so this is clean; cost: every call re-downloads (no cache), which is a bit wasteful for repeated calls.

③ 引入 xlrd>=2 运行时依赖③ Adds an xlrd>=2 runtime dependency

上游是旧式 .xls，只有 xlrd>=2（纯 Python、无二进制/编译）能读。符合 gget「纯 Python、pip 可装」的架构底线（不像 Rust/编译工具那样违背架构）。权衡：为单个模块给 core 加了一个运行时依赖；PR 作者已在描述里主动提议「可改成 optional extra」——建议采纳，与 gget[cellxgene] 的处理一致。The upstream is legacy .xls, and only xlrd>=2 (pure Python, no binary/compilation) can read it. This respects gget's architectural baseline of "pure Python, pip-installable" (unlike Rust/compiled tools, which would violate the architecture). Trade-off: it adds a runtime dependency to core for the sake of a single module; the PR author has already proactively proposed in the description that "it could be made an optional extra" — recommended to adopt, consistent with how gget[cellxgene] is handled.

④ URL 硬编码到个人目录④ URL hard-coded to a personal directory

直连 personal.broadinstitute.org/scalvo/MitoCarta3.0/*.xls。优点：版本固定 → 输出确定、可复现、可引用。最大风险：个人 web 目录无 SLA/无 DOI/无 API，一旦作者迁移或清理即 404 → 这是时间点快照式镜像，必须靠 live test 兜底发现腐烂。It connects directly to personal.broadinstitute.org/scalvo/MitoCarta3.0/*.xls. Pro: pinned version → deterministic, reproducible, citable output. Biggest risk: a personal web directory has no SLA / no DOI / no API — the moment the author moves or cleans it up it 404s → this is a point-in-time snapshot mirror, which must rely on a live test as a fallback to catch bit-rot.

按 skill 的 live-data 规范，现有测试需要三处加固：Per the skill's live-data spec, the existing tests need three reinforcements:
  1. 限流/网络错 → skip 而非红：现在上游 404/超时/返回 HTML 会让 CI 硬红（误报）。应捕获 requests.RequestException 与「返回非 Excel」时 self.skipTest(...)。rate-limit/network-error → skip rather than red: right now an upstream 404/timeout/returned-HTML turns CI hard red (a false alarm). It should catch requests.RequestException and, on a "non-Excel response," call self.skipTest(...).
  2. 精确行数 1136 太脆：应放宽为下限（如 >= 1000），把「身份锚」留给能对真值校验的断言——MT-CO1 ∈ Symbol、期望列存在、Content-Type 是 Excel 而非 HTML。这些锚已部分具备，方向对。the exact row count 1136 is too brittle: it should be loosened to a lower bound (e.g. >= 1000), leaving the "identity anchor" to assertions that can be checked against ground truth — MT-CO1 ∈ Symbol, expected columns present, Content-Type being Excel rather than HTML. These anchors are partly in place already; the direction is right.
  3. 轻量化 + 分层：源支持 Accept-Ranges: bytes，可先用 HEAD 探 Content-Type/Content-Length 做廉价探活；全量解析放 env-gate/nightly，并明确 log「只测了 human 一物种」。make it lightweight + layered: the source supports Accept-Ranges: bytes, so a HEAD probe of Content-Type/Content-Length can serve as a cheap liveness check; put the full parse behind an env-gate/nightly, and clearly log "only the human species was tested."
发布前应做一次负向验证（临时喂错文件名/错行数确认断言真的 fail），证明测试不是空测。Before release, do one negative verification (temporarily feed a wrong filename / wrong row count to confirm the assertions really fail), to prove the test is not a no-op.
