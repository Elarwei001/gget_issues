📖 背景科普Primer · 染色体名字的「巴别塔」The Babel of chromosome names
一个基因组（一个物种的全套 DNA）被切成若干条序列（sequence，通常就是一条染色体或一段脚手架）。麻烦在于：不同数据库给同一条染色体起的名字都不一样——人类 1 号染色体在 Ensembl 里叫 1、在 UCSC 浏览器里叫 chr1、在 NCBI 的 GenBank 里叫 CM000663.2、在 RefSeq 里叫 NC_000001.11。分析数据时如果两个文件用了不同命名体系，程序就会「对不上号」。NCBI 官方为每个基因组装配（assembly，即拼装好的一版参考基因组）发布一张装配报告（assembly report）——一张纯文本对照表，把同一条序列在四套命名体系下的名字并排列出。这个 PR 让 gget ref 能一键取回这张对照表。 A genome (a species' full DNA) is split into several sequences (usually one chromosome or scaffold each). The catch: different databases name the same chromosome differently — human chromosome 1 is 1 in Ensembl, chr1 in the UCSC browser, CM000663.2 in NCBI GenBank, and NC_000001.11 in RefSeq. When two files use different naming systems, tools fail to line them up. NCBI publishes an assembly report for each genome assembly (one assembled version of a reference genome) — a plain-text lookup table listing each sequence's name under all four conventions side by side. This PR lets gget ref fetch that table in one command.

3上游 / 数据源Upstream / data source

上游是 NCBI genomes FTP（https://ftp.ncbi.nlm.nih.gov/genomes/all/）。没有专门的查询 API——报告是静态文本文件（取 blob 型），但要拿到它得先解析 HTML 目录列表找到确切的文件夹名。路径规则：装配号 GCF_000001405.40 → 拆成 GCF / 000 / 001 / 405 / 四段目录 → 该目录下列出所有版本文件夹（如 GCF_000001405.40_GRCh38.p14/）→ 报告文件是 {folder}/{folder}_assembly_report.txt。PR 用 requests.get + BeautifulSoup 抓 <a> 链接、按 startswith(accession) 匹配文件夹名（因为装配号后面还跟着人读的装配名，如 _GRCh38.p14，事先不知道）。Upstream is the NCBI genomes FTP (https://ftp.ncbi.nlm.nih.gov/genomes/all/). There is no dedicated query API — the report is a static text file (blob-type), but reaching it requires scraping the HTML directory listing to find the exact folder name. Path rule: accession GCF_000001405.40 → split into GCF / 000 / 001 / 405 / → that directory lists all version folders (e.g. GCF_000001405.40_GRCh38.p14/) → the report file is {folder}/{folder}_assembly_report.txt. The PR uses requests.get + BeautifulSoup to grab <a> links and match the folder by startswith(accession) (because the accession is followed by a human-readable assembly name like _GRCh38.p14 that isn't known ahead of time).

8设计决策与权衡Design decisions & trade-offs

① 复用 gget ref 的位置参数，而非新命令Reuse the gget ref positional arg, not a new command

选择：把 species 位置参数在 assembly_report=True 时改解为装配号，并另暴露独立函数 gget.assembly_report()。好处：不新增顶层命令、与「取参考基因组」语义相邻。代价：一个参数身兼两义（物种名 vs 装配号），靠开关切换，稍有认知负担；用 _assembly_report_fn 别名巧妙避开了布尔参数与函数同名的遮蔽问题。Choice: reinterpret the species positional arg as an accession when assembly_report=True, and also expose a standalone gget.assembly_report(). Upside: no new top-level command, semantically adjacent to fetching references. Cost: one arg carries two meanings (species vs accession) toggled by a flag — slight cognitive load; a _assembly_report_fn alias neatly avoids the boolean param shadowing the same-named function.

② CLI --csv 旗标语义反了The CLI --csv flag is inverted

CLI 默认输出就是 CSV；加 --csv 反而输出 JSON（内部 json=args.csv）。名字与行为相反，几乎肯定会让用户困惑。建议改名为 --json（或默认 JSON、用 --csv 切 CSV）。这是本 PR 最该修的表层问题。CLI output is CSV by default; passing --csv instead emits JSON (internally json=args.csv). The name contradicts the behavior and will almost certainly confuse users. Suggest renaming to --json (or default to JSON and use --csv to switch to CSV). This is the most worth-fixing surface issue in the PR.

③ HTML 目录抓取而非拼死链接Scrape the directory instead of hard-coding the link

因为报告文件名里含事先未知的人读装配名（_GRCh38.p14），无法只凭装配号拼出完整 URL，所以必须先列目录再匹配。这是对上游现实的合理妥协，但也正是维护成本的来源（格式一变就得改解析）。Because the report filename embeds an unknown human-readable assembly name (_GRCh38.p14), the full URL can't be built from the accession alone — hence listing the directory first. A reasonable concession to upstream reality, but also the source of maintenance cost (a format change breaks the parse).
