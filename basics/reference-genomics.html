1转录本 (transcript) 是什么What is a transcript

转录本 = 基因被"抄录"出来的那份 RNA 拷贝。细胞不敢直接动用 DNA 原本，而是先抄一份 RNA 副本拿去用——这个"抄"的动作叫转录 (transcription)，抄出来的产物就是转录本。整条流水线是：A transcript is the RNA copy made from a gene. A cell won't use the DNA original directly; it first makes an RNA copy to work from — that copying act is transcription, and its product is the transcript. The full pipeline is:

DNA  --(转录 transcription)-->  转录本 RNA transcript  --(翻译 translation)-->  蛋白质 protein

关键点：一个基因能抄出多个不同的转录本Key point: one gene can yield several different transcripts

抄图时可以只抄某几段、跳过另几段，拼出不同版本的副本——这叫可变剪接 (alternative splicing)。所以一个基因可能对应多个转录本，进而可能造出多种略有差异的蛋白。这也是人类约 2 万个基因却能产生远多于 2 万种蛋白的原因。When copying, the cell can keep only certain segments and skip others, producing different versions — this is alternative splicing. So one gene can map to many transcripts, and hence to several slightly different proteins. That's why ~20,000 human genes can produce far more than 20,000 proteins.

连回 gget：gget ref -w cdna 取的"transcript sequences（转录本序列）"就是这些 RNA 副本的序列集合；-w ncrna 取的是 lncRNA 转录本——一类抄出来但不拿去造蛋白、自身另有调控作用的特殊转录本（long non-coding RNA，长非编码 RNA）。Back to gget: gget ref -w cdna fetches "transcript sequences" — the sequences of these RNA copies; -w ncrna fetches lncRNA transcripts — a special class that is copied out but not used to build protein, instead playing regulatory roles (long non-coding RNA).

2RNA-seq 是什么What is RNA-seq

一句话：测一测细胞此刻正在"抄录"哪些基因、各抄了多少份。In one line: measure which genes a cell is currently transcribing, and how many copies of each.

同一套设计图，不同细胞、不同状态下抄的图和份数都不同：肝细胞和神经元开的基因完全不一样；癌细胞会疯狂多抄某些图。RNA-seq（RNA 测序）就是把细胞里所有 RNA 副本捞出来 → 用测序仪逐条读出序列 → 统计每个基因/转录本被抄了多少份。得到的这张"各基因表达量"清单叫基因表达谱。The same blueprint set is copied differently across cells and states: liver cells and neurons switch on entirely different genes; cancer cells over-copy certain blueprints. RNA-seq (RNA sequencing) pulls out all the RNA copies in a cell → reads each one on a sequencer → counts how many copies of each gene/transcript there are. The resulting "expression per gene" list is the gene expression profile.

🏭 不看固定的设计图库（DNA），而是去车间清点今天实际在用哪些副本、各印了多少张。用途：比较 病 vs 健康、用药前 vs 用药后、这种细胞 vs 那种细胞——看谁的基因开多了/关掉了。🏭 Don't look at the fixed blueprint archive (DNA); go to the floor and count which copies are actually in use today and how many of each. Uses: disease vs healthy, before vs after a drug, one cell type vs another — to see which genes went up or down.

3比对与索引：kallisto / STAR index 是什么Alignment & index: what kallisto / STAR index means

测序仪读出来的是几千万条几十~上百字母的短片段（叫 reads），但它们不带标签——你不知道每条来自哪个基因。必须先把每条 read"归位"到参考序列上，才能数数。A sequencer outputs tens of millions of short fragments of tens-to-hundreds of letters (called reads), unlabeled — you don't know which gene each came from. You must first map each read back onto the reference before you can count.

  • STAR、kallisto 是做这件事的软件：STAR 做比对 (alignment)，kallisto 做更快的伪比对 (pseudo-alignment)。STAR and kallisto are the tools for this: STAR does alignment, kallisto does a faster pseudo-alignment.
  • 索引 (index) = 这些工具为了快速查找，预先把参考序列建成的一张"字典/目录"。The index is a "dictionary/lookup table" these tools pre-build from the reference so lookups are fast.
📖 就像给一本没有目录的大词典先编好索引：之后查任何词都能秒定位，而不必每次从头翻。建索引是"一次性预处理"，之后每条 read 归位都很快。📖 Like adding an index to a dictionary that had none: afterwards any word is found instantly, instead of scanning from page one every time. Building the index is a one-time preprocessing step that makes every subsequent read lookup fast.
连回 gget：建索引的输入正是 gget ref 取的参考文件（转录本 FASTA + GTF）。所以链路是：gget ref 下载参考 → kallisto/STAR 建索引 → 拿 RNA-seq 的 reads 去数每个转录本多少条。gget 文档里的 gget ref … | kb ref 示例就是这个衔接。Back to gget: the input to building an index is exactly the reference files gget ref fetches (transcript FASTA + GTF). So the chain is: gget ref downloads the reference → kallisto/STAR builds the index → RNA-seq reads are counted against each transcript. The gget ref … | kb ref example in gget's docs is precisely this hand-off.

对应到 gget ref 的 which：Mapped to gget ref's which:

  • -w dna → genome FASTA：整条染色体的字母串（几十亿个）。genome FASTA: the letter string of whole chromosomes (billions of them).
  • -w cdna → transcript FASTA：只把每个转录本拼出来的序列。transcript FASTA: just the assembled sequence of each transcript.
  • -w gtf → GTF：不含字母，只含"谁在哪儿"的坐标表。GTF: no letters, only the "who sits where" coordinate table.
为什么两个都要？光有字母（FASTA）不知道哪段是基因；光有坐标（GTF）不知道具体序列。合起来才能"从整条染色体上、按 GTF 的坐标、切出某基因的序列"。Why both? With only letters (FASTA) you can't tell which stretch is a gene; with only coordinates (GTF) you don't have the actual sequence. Together they let you "cut out a gene's sequence from the whole chromosome, at the coordinates GTF specifies".

5Ensembl 和 GENCODE：既然都有转录本，为什么还要两个？Ensembl vs GENCODE: both have transcripts, so why both?

核心答案：对人和小鼠，GENCODE 与 Ensembl 的注释其实"同源"——同一批人协同产出、内容高度重叠——但它们不是同一份文件，发布节奏、打包方式、ID 习惯、社区默认都不同，所以要能分别取。Core answer: for human and mouse, GENCODE and Ensembl annotations are essentially co-produced by the same groups and overlap heavily — but they are not the same file. Release cadence, packaging, ID conventions, and community defaults all differ, so you need to fetch each.

  1. 同源但不等同：人/鼠的 GENCODE 与 Ensembl 注释由同一团队（EBI/Havana + Ensembl）协同产生，内容高度重叠；但 GENCODE 有自己的 release 节奏、会额外发布一些资源，版本号体系也不同（GENCODE v46 ≠ Ensembl 110）。Co-produced, not identical: human/mouse GENCODE and Ensembl annotations come from the same teams (EBI/Havana + Ensembl) and overlap heavily; but GENCODE has its own release cadence, publishes some extra resources, and uses a different version scheme (GENCODE v46 ≠ Ensembl 110).
  2. 社区默认不同（最实际的原因）：很多主流数据集和流程默认认 GENCODE——如 GTEx、TCGA、10x/Cell Ranger 的多数参考。你的分析若要和这些数据对齐，就必须用 GENCODE 那份，不能拿 Ensembl 替；反之另一些工具默认 Ensembl。混用两套注释会导致基因 ID/坐标对不上。Different community defaults (the most practical reason): many mainstream datasets and pipelines default to GENCODE — e.g. GTEx, TCGA, and most 10x/Cell Ranger references. If your analysis must align with those, you must use the GENCODE one, not Ensembl; conversely other tools default to Ensembl. Mixing the two annotations causes gene IDs/coordinates to mismatch.
  3. 打包/命名习惯不同：染色体命名（chr1 vs 1）、文件组织、ID 版本号写法等细节不同，下游工具往往挑食。Different packaging/naming: chromosome names (chr1 vs 1), file organization, ID-versioning style — details differ, and downstream tools are often picky.
  4. 可复现性：论文里写"用 GENCODE v46"是明确、可复跑的；换成 Ensembl 就是另一份输入了。Reproducibility: "used GENCODE v46" in a methods section is precise and re-runnable; swapping in Ensembl is a different input.
一句话：不是"一样的东西取两遍"，而是"两套虽同源、但各有默认场景的标准，得都能取"。所以 gget 加 GENCODE 不是重复造轮子，而是补上另一套社区标准——gget ref … --source ensembl|gencode 让你精确取到需要的那一份。In one line: it's not "fetching the same thing twice", but "two co-produced standards, each the default in different settings — you must be able to fetch either". So adding GENCODE to gget isn't reinventing a wheel; it fills in the other community standard — gget ref … --source ensembl|gencode gets you exactly the one you need.

6串起来：一条典型链路Putting it together: a typical chain

1. gget ref  --source gencode -w cdna,gtf human      # 取参考：转录本 FASTA + GTF
2. kb ref  /  STAR --runMode genomeGenerate           # 建索引（给参考编"目录"）
