8设计决策与权衡Design decisions & trade-offs

① 一个 resource 参数分派三类查询One resource param dispatching three query types

用 resource="pathways"|"search"|"entity" 在一个模块里分派三个端点,而非三个独立命令。合理——它们同属"查 Reactome",共享 query/species 等参数,一个心智模型;和 opentargets 的 resource 风格一致。Dispatches three endpoints in one module via resource="pathways"|"search"|"entity" rather than three separate commands. Reasonable — they're all "query Reactome", share query/species params, one mental model; consistent with opentargets' resource style.

② 把"不整齐"的上游返回规整成固定列Normalizing uneven upstream responses to fixed columns

剥 HTML 高亮标签、扁平 species 列表、displayName→name[0] 回退——这些是把 ContentService 略显异构的 JSON 变成 tidy 表的必要工作,体现了 gget "把数据源统一成一致接口"的价值。Stripping HTML highlight tags, flattening the species list, displayName→name[0] fallback — these turn ContentService's slightly heterogeneous JSON into tidy tables, embodying gget's "unify sources into a consistent interface" value.

③ 404 当"无结果"而非报错(仅 entity 例外)404 as "no results" (except entity)

pathways/search 的 404 → 返回空 DataFrame(合理:查无结果是正常态);entity 的 404 → ValueError(合理:给了个无效 stable ID 是用户错误)。这个区分是对的。实现上靠字符串匹配错误文案,建议未来让 http_json 暴露结构化状态码更稳。pathways/search 404 → an empty DataFrame (reasonable: no results is a normal outcome); entity 404 → a ValueError (reasonable: a bad stable ID is user error). This distinction is right. It's implemented via string-matching the error message; a sturdier future option is to have http_json expose a structured status code.
